## 1. 系统概述

### 1.1 项目背景

粮库作业环境中，工作人员需佩戴安全帽、呼吸机等防护装备，同时需防范人员聚集、打架斗殴、摔倒等突发事件。传统人工巡检方式存在覆盖范围有限、响应延迟高、夜间盲区等问题。本系统利用**算能BM1684边缘计算盒子**的TPU算力，结合**海康威视网络摄像头**，构建了一套完整的AI视频分析监控闭环。

### 1.2 系统能力

| 检测类型 | YOLO模型 | 目标类别 | 触发条件 |
|---------|----------|---------|---------|
| 安全帽检测 | helmet | head (0), helmet (1) | 头部未佩戴安全帽持续≥1.0s |
| 呼吸机检测 | gasmask | head, mask, tank | 面罩/氧气瓶缺失≥2.0s |
| 摔倒检测 | fall | person | 人员中心点快速下移+宽高比突变 |
| 打架检测 | fight | person | 多人肢体交互区域检测 |
| 人员聚集检测 | crowd | person (复用helmet模型) | DBSCAN聚类≥3人且持续稳定 |
| 门窗检测 | windows | door_open, door_close, window_open, window_close | 门/窗状态变化检测 |

### 1.3 关键性能指标

- **推理延迟：** 单模型batch=4典型total ~80-90ms（preprocess ~66-76ms，真正TPU推理 ~6-10ms）
- **端到端延迟：** 视频采集到告警触发 < 500ms
- **并发路数：** 稳定支持4路RTSP并发（架构支持扩展至9路）
- **预览帧率：** MJPEG 20fps目标（四宫格合成显示）
- **TPU利用率：** 批调度优化后TPU持续工作，利用率取决于模型数量和batch size
- **告警录制：** 预录6s + 后录6s，使用硬件H.264编码

### 2.2 多进程架构

系统采用**多进程 + 共享内存**架构，核心进程划分如下：

| 进程 | 角色 | 通信方式 |
|------|------|---------|
| **Main** (Flask) | Web服务器、任务调度入口 | - |
| **DecodeHub** | RTSP拉流解码、帧分发 | SharedMemory + mp.Queue |
| **ModelScheduler** | 批量推理调度、模型管理 | SharedMemory (FrameHub) |
| **DisplayService** | 预览合成、告警渲染、报警录制 | mp.Queue (result/display) |

**关键设计决策——多进程而非多线程：**

由于BM1684的SAIL库在TPU推理时涉及设备内存管理和BMCV硬件加速操作，多线程模式在GIL和对象生命周期管理上存在潜在风险。采用多进程架构可以实现：
- 解码进程崩溃不影响推理进程
- 推理进程可独立重启
- 各进程通过共享内存传递帧数据，避免大图像数据跨进程序列化开销

### 2.3 帧数据共享机制——LatestFrameHub

`core/frame_hub.py` 实现了一个**多进程安全的帧共享中心**，核心技术：

```python
# 每路视频一个固定大小的共享内存槽位
- 图像数据: multiprocessing.shared_memory.SharedMemory (BGR uint8, max 1920x1080x3)
- 元数据: multiprocessing.Array (ctypes.c_longlong) — frame_number, width, height, channels, generation等
- 时间戳: multiprocessing.Array (ctypes.c_double)
- 互斥锁: multiprocessing.Lock (每路独立)
```

**关键机制：**
- **Generation计数器：** 每次写入递增，消费者对比generation判断是否有新帧
- **Slot预先缩放：** DecodeHub写入前将检测帧缩放到640x360（由`detect_frame_max_width/height`配置），减少共享内存占用和后续预处理开销
- **coord_width/coord_height：** 保留检测坐标系尺寸字段，确保报警框坐标能正确映射回原始分辨率

## 3. 视频采集与编解码

### 3.1 解码架构

系统支持两种解码模式，由`decode_backend`配置控制：

#### 3.1.1 MultiDecoder模式（推荐）

```python
# ingestion/decode_hub.py
self._multi_decoder = sail.MultiDecoder(16, self.device_id)  # 最多16通道
self._multi_decoder.set_local_flag(True)  # 本地时间戳，降低延迟
for stream_id, spec in enumerate(self.source_specs):
    self._multi_channels[stream_id] = self._multi_decoder.add_channel(spec["source"], 0)
```

**技术优势：**
- BM1684硬件解码器，RTSP流直接解码到BMImage（设备内存）
- 避免CPU→TPU数据搬运
- 多路流在一个轮询循环中并发读取
- 支持RTSP over TCP，自动重连

#### 3.1.2 DecoderPool模式（兼容方案）

当设备不支持MultiDecoder或输入为本地视频文件时，回退到逐路独立解码器：

```python
# 单路解码器
self.decoder = sail.Decoder(self.source, True, self.device_id)  # 硬件解码
# 或回退
self.cap = cv2.VideoCapture(self.source)  # OpenCV软件解码
```

**自适应回退链：** sail.Decoder → cv2.VideoCapture（带FFmpeg低延迟参数）

### 3.2 解码→检测→预览分流

DecodeHub内部采用**检测/预览双时钟独立抽帧**策略：

```python
# 检测侧: 低帧率抽帧 (detect_emit_fps=3, 即每路每秒3帧)
detect_interval = 1.0 / float(self.detect_emit_fps)  # ~333ms
# 预览侧: 高帧率抽帧 (preview_source_fps=20)
preview_interval = 1.0 / float(self.preview_source_fps)  # 50ms
```

**两路独立缩放：**

| 链路 | 缩放目标 | 后端 | 用途 |
|------|---------|------|------|
| 检测帧 | 640×360 (max) | cv2.resize | 写入FrameHub供模型推理 |
| 预览帧 | 320×180 (四宫格) / 1280×720 (单路) | bmcv.resize 或 cv2 | 写入预览Queue供Display合成 |

**PreviewScaler——自适应缩放后端选择：**

```python
class PreviewScaler:
    def __init__(self, device_id, requested_backend, target_width, target_height):
        if requested_backend == "bmcv" and SOPHON_AVAILABLE:
            self.handle = sail.Handle(device_id)
            self.bmcv = sail.Bmcv(self.handle)     # 硬件缩放
        else:
            self.actual_backend = "cv2"              # 软件缩放（避免与推理抢BMCV）
```

默认配置为`scale_backend=cv2`，将BMCV资源留给推理链路，预览缩放使用CPU完成。

### 3.3 RTSP连接管理

每路通道维护连接状态，断线自动重连：

```python
class _DecoderPoolChannel:
    def reconnect(self):
        if self.is_video_file:  # 本地视频文件直接重连
            return self.connect()
        now = time.monotonic()
        if now - self.last_reconnect_ts < 1.0:  # 重连间隔≥1s
            return False
        self.last_reconnect_ts = now
        return self.connect()
```

RTSP推流端（海康摄像头）配置TCP传输以增强稳定性：
```python
os.environ["OPENCV_FFMPEG_CAPTURE_OPTIONS"] = \
    "rtsp_transport;tcp|fflags;nobuffer|flags;low_delay|max_delay;0"
```

| fall_detection | fall_yolo26s_640_int8_1b.bmodel | 人员摔倒检测 | 1 (person) | 640×640 |

### 4.2 BM1684X_YOLO26推理核心

`core/bm1684x_yolo26_adapter.py` 是本系统的推理核心，管理TPU推理的完整生命周期。

#### 4.2.1 模型加载与初始化

```python
def _load_model(self):
    # 创建推理引擎 (SYSO模式: 系统内存与设备内存分离)
    self.engine = sail.Engine(self.bmodel_path, self.device_id, sail.IOMode.SYSO)
    self.handle = sail.Handle(self.device_id)
    self.bmcv = sail.Bmcv(self.handle)

    # 获取模型图信息
    self.graph_name = self.engine.get_graph_names()[0]
    self.input_name = self.engine.get_input_names(self.graph_name)[0]

    # 获取输入规格
    self.input_shape = self.engine.get_input_shape(self.graph_name, self.input_name)
    # shape: [batch_size, 3, net_h, net_w]  例如 [4, 3, 640, 640]

    # 预分配输出Tensor（reusable）
    for output_name in self.output_names:
        self.output_tensors[output_name] = sail.Tensor(
            self.handle, output_shape, output_dtype, True, True
        )
```

#### 4.2.2 输入张量复用（关键优化）

这是本系统最核心的优化之一。每次推理不再重新创建输入Tensor和BMImageArray，而是在初始化时预分配并复用：

```python
def _init_reusable_input_buffers(self):
    # 主输入Tensor (batch_size × C × H × W)
    self._input_tensor = sail.Tensor(self.handle, self.input_shape, self.input_dtype, True, True)

    if self.batch_size > 1:
        # 批量BMImageArray (将batch个BMImage的数据指针打包)
        self._bm_image_array_type = getattr(sail, f"BMImageArray{self.batch_size}D", None)
        if self._bm_image_array_type is not None:
            self._bm_image_array = self._bm_image_array_type()
        else:
            # Fallback: 当SAIL不支持特定batch size时（如BMImageArray9D）
            # 使用8D + 单帧混合策略
            self._fallback_bm_image_array = BMImageArray8D()  # 前8帧
            self._fallback_tensor = Tensor([8, C, H, W])
            # 第9帧单独处理
            self._single_image_tensors[8] = Tensor([1, C, H, W])
```

**为什么需要混合策略？** 实际测试发现SAIL库缺少`BMImageArray9D`类型定义。当batch=9且没有原生9D数组类型时，系统自动降级为8D+1D混合路径——前8帧通过BMImageArray8D批处理，第9帧单独通过单帧Tensor处理，然后通过numpy拼接合并后再一次性sync到设备。

#### 4.2.3 预处理流水线（BMCV硬件加速）

预处理是将原始BGR图像转换为模型输入格式的完整流水线，全部使用BMCV硬件加速：

```python
def preprocess_bmcv(self, image: np.ndarray, slot_index=None):
    """
    预处理流水线（全部BMCV硬件操作）：
    1. numpy → BMImage (BGR_PACKED)        — mat_to_bm_image
    2. BGR_PACKED → RGB_PLANAR             — convert_format
    3. Resize + Letterbox Padding           — crop_and_resize_padding
    4. 归一化 (mean/std → scale+bias)       — convert_to
    """
    # Step 1-2: 格式转换
    self.bmcv.mat_to_bm_image(image, bmimg_bgr)
    self.bmcv.convert_format(bmimg_bgr, rgb_planar_img)

    # Step 3: Letterbox resize with padding
    # 计算等比缩放参数
    ratio = min(net_w / img_w, net_h / img_h)
    tw, th = round(ratio * img_w), round(ratio * img_h)
    tx1, ty1 = (net_w - tw) / 2.0, (net_h - th) / 2.0

    # BMCV硬件resize + padding (padding值=114灰色)
    attr = sail.PaddingAtrr()
    attr.set_stx(int(round(tx1 - 0.1)))
    attr.set_sty(int(round(ty1 - 0.1)))
    attr.set_w(tw); attr.set_h(th)
    attr.set_r(114); attr.set_g(114); attr.set_b(114)

    resized_img_rgb = self.bmcv.crop_and_resize_padding(
        rgb_planar_img, 0, 0, img_w, img_h,
        net_w, net_h, attr, sail.bmcv_resize_algorithm.BMCV_INTER_LINEAR
    )

    # Step 4: 归一化 (scale=1/255, 存入int8/float16格式)
    # ab = [α₀, β₀, α₁, β₁, α₂, β₂] 对应RGB三通道
    self.bmcv.convert_to(resized_img_rgb, preprocessed_bmimg,
        ((self.ab[0], self.ab[1]), (self.ab[2], self.ab[3]), (self.ab[4], self.ab[5])))
```

**预处理槽位复用（Preprocess Slots）：**
每次预处理还需要中间BMImage对象。通过按输入尺寸索引的槽位缓存，只要后续帧尺寸不变，直接复用：
```python
def _get_preprocess_slot(self, slot_index, img_h, img_w):
    slot = self._preprocess_slots[slot_index]
    if slot.get("src_shape") != (img_h, img_w):
        # 尺寸变化时才重建
        slot["bgr"] = sail.BMImage(...)
        slot["rgb"] = sail.BMImage(...)
        slot["preprocessed"] = sail.BMImage(...)
    return slot
```

#### 4.2.4 Batch推理与后处理

```python
def infer(self, images: list[np.ndarray]):
    # 1. 批量预处理（BMCV操作，当前串行执行——优化空间）
    preprocessed, original_sizes, ratios, txy_list = self._preprocess_images(images)

    # 2. 构建输入Tensor
    input_tensor = self._build_input_tensor(preprocessed)

    # 3. TPU推理（核心瓶颈仅6-10ms）
    self.engine.process(self.graph_name, input_tensors, input_shapes, output_tensors)

    # 4. 输出读取 & 反量化
    output = self.output_tensors[detection_output_name].asnumpy()
    output = output.astype(np.float32) * output_scale  # INT8→FP32反量化

    # 5. 后处理：坐标反算（从letterbox空间→原始图像空间）
    # 公式: x_orig = (x_pred - tx1) / ratio
    #       y_orig = (y_pred - ty1) / ratio
    detections[:, [0,2]] = (detections[:, [0,2]] - tx1) / ratio[0]
    detections[:, [1,3]] = (detections[:, [1,3]] - ty1) / ratio[1]
    detections[:, [0,2]] = np.clip(detections[:, [0,2]], 0, org_w - 1)
    detections[:, [1,3]] = np.clip(detections[:, [1,3]], 0, org_h - 1)
```

## 5. 批调度与分时推理

### 5.1 BatchModelScheduler架构

`app/pipeline/model_scheduler.py` 实现了**多模型、多通道的批量推理调度器**，核心职责：

1. 管理模型生命周期（加载/卸载）
2. 按配置的时间间隔调度各模型推理
3. 收集多路帧组装batch
4. 将推理结果分发给算法消费者

- `helmet_detection`：检测人员头部(head)
- 算法层将面罩/氧气瓶与最近的人头做关联匹配

### 5.3 轮询调度与时间间隔控制

```python
def _due_models(self) -> list[str]:
    """按轮询顺序返回当前到期的模型"""
    now = time.monotonic()
    active = self._active_model_keys()
    # Round-Robin起始偏移
    ordered = active[self._rr_index % len(active):] + active[:self._rr_index % len(active)]
    due = []
    for model_key in ordered:
        interval = self.model_intervals.get(model_key, 0.2)
        if now - self.last_model_infer_ts.get(model_key, 0.0) >= interval:
            due.append(model_key)
    return due
```

### 5.5 FrameContext与LatestFrameCache

`app/pipeline/time_share.py` 实现了基于时间的帧缓存，确保每次推理拿到的是该路最新的帧：

```python
class LatestFrameCache:
    def pick_due_streams(self, enabled_streams) -> list[int]:
        """返回超过min_update_interval_s未推理的通道"""
        now = time.monotonic()
        due = []
        for sid in enabled_streams:
            last = self._last_inferred.get(sid, 0.0)
            if now - last >= self.min_update_interval_s:
                due.append(sid)
        return due
```

### 6.1 算法架构

每个检测算法继承自`DetectionAlgorithm`基类，实现`process(frame_context, inference_outputs, state)`接口：

```python
class DetectionAlgorithm:

            frame_context: 帧上下文 (帧数据、帧号、时间戳等)
            inference_outputs: 模型推理结果 (DetectionBox列表)
            state: 算法持久状态字典 (跨帧维护)
        Returns:
            AlgorithmResult: 包含检测详情、显示告警、可记录告警
        """
```

### 6.2 基于事件状态机的告警决策

`app/algorithms/support/event_state.py` 实现了**滑动窗口事件状态机**，用于将单帧检测结果转换为稳定的告警决策：

**核心机制——时间窗口 + 命中比例：**

```python
class TrackEventStateMachine:
    """
    基于滑动时间窗口的事件判定：
    - window_seconds: 观察窗口大小（如安全帽：1.0s）
    - threshold_ratio: 触发阈值（如未佩戴率 > 70%）
    - min_observation_seconds: 最小观察时长（避免不足窗口即报警）
    - cooldown_seconds: 冷却时间（同一目标报警后不再重复报警）

    状态转移：
    IDLE → OBSERVING（首次命中）→ ALERTED（命中率超阈值且观察时长达标）
                                    ↓
                                COOLDOWN（冷却期后回到IDLE）
    """
```

**实例1：安全帽检测（HelmetDetectionAlgorithm）**

```python
class HelmetDetectionAlgorithm:
    # 核心逻辑：
    # 1. SimpleIOUTracker 跟踪每个人头目标 (匈牙利匹配/IOU+中心距离)
    # 2. 对每个跟踪目标，累计 "未佩戴安全帽" 的持续时间
    # 3. 当某目标在1.0s观察窗口内未佩戴率 > 70%，且观察时长 ≥ 1.0s → 触发告警
    # 4. 告警后冷却180s（同一目标）
```

**实例2：摔倒检测（FallDetectionAlgorithm）**

```python
# 核心逻辑：
# 1. 检测人体bbox
# 2. 计算bbox的宽高比 (aspect_ratio = width/height)
#    站立: aspect_ratio ≈ 0.3-0.6
#    摔倒: aspect_ratio ≈ 1.5-3.0
# 3. 1.5s观察窗口内 "摔倒状态帧" ≥ 50% 且观察时长 ≥ 1.0s → 触发告警
```

**实例3：人员聚集检测（CrowdDetectionAlgorithm）**

```python
# 核心逻辑：
# 1. 提取所有人体的中心点坐标
# 2. DBSCAN聚类 (eps=50px, min_samples=3)
# 3. 当某个聚类≥3人且连续1.0s内稳定（稳定性比例 ≥ 60%）→ 触发告警
# 4. cooldown仅30s（聚集事件更频繁）
```

### 7.2 VideoBufferManager——预录+异步保存

```python
class VideoBufferManager:
    def __init__(self, fps=30, pre_alarm_seconds=6, post_alarm_seconds=6):
        # 环形预录缓冲
        self.pre_buffer = deque(maxlen=pre_alarm_seconds * 30)  # 最多180帧

        # 后台保存线程（不阻塞主渲染循环）
        self.save_queue = Queue(maxsize=10)
        self.save_thread = threading.Thread(target=self._async_save_worker, daemon=True)
        self.save_thread.start()

    def trigger_alarm(self, alarm_id, alarm_type, alarm_info, stream_id):
        """触发告警：从预缓冲中提取触发前6s的帧，标记开始后录"""
        cutoff_time = current_time - timedelta(seconds=self.pre_alarm_seconds)
        pre_frames = [f for f in self.pre_buffer if f['timestamp'] >= cutoff_time]
        self.recording_alarms[alarm_id] = {
            'frames': pre_frames,
            'end_time': current_time + timedelta(seconds=self.post_alarm_seconds)
        }

    def update(self, frame, frame_number, timestamp):
        """每帧update：对正在录制的告警追加帧"""
        for alarm_id, recording in self.recording_alarms.items():
            recording['frames'].append(frame_data)
            if timestamp >= recording['end_time']:
                self._queue_save_task(alarm_id)  # 异步保存，不阻塞
```

### 8.4 基于光流的报警框跟踪

在两次检测结果之间（检测为低频3fps，预览为20fps），通过**LK光流法**跟踪报警框位置：

```python
def _track_alert_box_with_flow(self, prev_gray, gray, alert, frame_shape, now):
    # 1. 在报警框ROI内提取角点特征 (goodFeaturesToTrack)
    points = cv2.goodFeaturesToTrack(roi, maxCorners=18, qualityLevel=0.01, minDistance=3)

    # 2. 前向光流追踪 (calcOpticalFlowPyrLK)
    next_points, status, _ = cv2.calcOpticalFlowPyrLK(prev_gray, gray, points, ...)

    # 3. 后向光流校验（Forward-Backward Error）
    back_points, back_status, _ = cv2.calcOpticalFlowPyrLK(gray, prev_gray, next_points, ...)
    fb_err = np.linalg.norm(back_points - points, axis=1)
    valid = valid & (fb_err < 2.0)  # 过滤前后向不一致的点

    # 4. 中值位移 + 位移限制
    med_x, med_y = np.median(shifts[:, 0]), np.median(shifts[:, 1])
    dx, dy = self._limit_alert_shift(bbox, med_x, med_y, max_shift_ratio=0.35)
    shifted_bbox = [x1+dx, y1+dy, x2+dx, y2+dy]
```

这样在检测结果更新间隙，报警框也能平稳跟随目标，避免"框跳来跳去"的观感问题。

## 10. 性能优化策略

### 10.1 优化历程总结

| 阶段 | 优化内容 | 效果 |
|------|---------|------|
| **基线** | 9路全开，原始链路 | preprocess ~100-140ms, total ~120-150ms |
| **阶段1** | YOLO26 input tensor/BMImageArray复用 | preprocess ~88-98ms, total ~96-109ms |
| **阶段2** | 检测帧缩放到640x360 + 8D+1D混合Fallback | preprocess ~66-76ms, total ~80-90ms |
| **阶段3** | 预览/检测资源隔离 (scale_backend=cv2) | 检测吞吐不受浏览器预览影响 |
| **阶段4** | 调度参数精细调优 (频率/interval/batch) | TPU利用率提升，结果延迟下降 |

### 10.2 关键优化技术点

**a) Tensor/BMImageArray复用**
- 预创建输入Tensor、输出Tensor、BMImageArray
- 避免每次推理分配/释放设备内存
- 不足batch时用最后一帧填充（保持batch size恒定）

**b) 预处理槽位缓存**
- 按源尺寸索引缓存中间BMImage对象
- 同一路视频尺寸不变时零分配

**c) 检测帧提前缩小**
- DecodeHub写入FrameHub前缩放到640x360
- 减少共享内存占用（1920×1080×3 ≈ 6MB → 640×360×3 ≈ 0.7MB）
- 减少后续BMCV resize的计算量

**d) 8D+1D混合Batch策略**
- 当SAIL缺少某batch size的BMImageArray类型时
- 前N帧用最大可用chunk批量处理，剩余帧单独处理
- numpy拼接后统一sync_s2d（一次设备数据传输）

**e) 预览/检测资源隔离**
- 检测链路使用BMCV硬件缩放
- 预览链路使用cv2软件缩放（CPU）
- 避免两个链路竞争BMCV导致互相阻塞

**f) 异步告警保存**
- 视频编码和写入在独立线程执行
- 主渲染循环不等待IO完成
- 保存队列满时丢弃而非阻塞（告警不丢，保存可丢）

## 12. 系统特性总结

### 12.1 技术亮点

1. **边缘计算全链路闭环：** 从视频采集、BM1684硬件解码、BMCV预处理、TPU推理、算法分析到告警生成，全流程在边缘盒子完成，无需云端依赖

2. **INT8量化模型高效推理：** YOLO26模型经TPU-MLIR工具链量化后，单帧推理仅6-10ms（TPU时间），配合输入复用和预处理优化，batch=4的total控制在80ms以内

3. **多进程共享内存架构：** 解码、推理、展示三大进程通过LatestFrameHub共享内存交换帧数据，避免大图像数据序列化开销，实现低延迟帧传递

4. **批调度与分时推理：** Round-Robin轮询多模型调度，每模型独立时间间隔控制，支持模型共享（多个算法复用同一模型输出），最大化TPU利用率

5. **事件状态机告警决策：** 基于滑动时间窗口+命中比例的告警判定，配合冷却机制，有效抑制单帧误检导致的告警风暴

6. **光流跟踪报警框平滑：** 在检测间隔（低频3fps）内使用LK光流+前后向误差校验跟踪报警框位置，实现20fps预览下的平滑显示

7. **硬件编码异步保存：** 告警视频使用BM1684硬件H.264编码器，保存操作在后台线程异步执行，不影响主渲染循环

8. **检测帧缩放与坐标保持：** 检测帧提前缩小到640×360减少计算量，同时通过coord_width/coord_height保持坐标映射精度

### 12.2 实际应用价值

- **低延迟：** 端到端延迟 < 500ms，满足实时安全监管需求
- **本地化部署：** 无需云服务，数据不出局域网，符合粮库信息安全要求
- **可扩展性：** 架构支持扩展到更多路视频流、更多检测算法
- **鲁棒性：** 解码进程崩溃可自动重连，多进程隔离故障不扩散
- **易用性：** Web界面操作，手机/电脑均可访问，无需安装客户端
